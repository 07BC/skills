## Audit: Chagi/Shared/Player/VODPlayerViewModel.swift

**Surface:** `handleBackground()` and `handleForeground()` modified to gate background resume on a new `wasPlayingBeforeBackground` flag.

### 🔴 BLOCKER
- `VODPlayerViewModel.swift:155-167` — The KVO observer on `\.currentItem?.status` calls `player.play()` unconditionally whenever the item transitions to `.readyToPlay`. On return from background the AVPlayerItem typically re-stabilises (e.g. `.unknown` → `.readyToPlay`, or the decoder/network session is re-established), which fires this observer and restarts playback regardless of the new `wasPlayingBeforeBackground` guard added in `handleForeground()`. The fix is incomplete — the bug originates in this observer, not in the lifecycle handlers. Either gate the auto-play on `wasPlayingBeforeBackground` (and `isPaused` for the in-session pause case), or remove the unconditional `player.play()` from the status observer and trigger initial playback explicitly from `loadPlaybackUrl`.

### 🟡 RISK
- `VODPlayerViewModel.swift:216` — `wasPlayingBeforeBackground` is a plain `Bool` on a non-isolated `NSObject` subclass. Step 4 trap: KVO callbacks on AVPlayer are not guaranteed to fire on the main thread. The `timeControlStatus` observer at line 170 currently only writes `isPaused`/`hidePauseImage`, but if any future change reads `wasPlayingBeforeBackground` from inside a KVO callback (a natural next step if the BLOCKER above is fixed by gating the observer), the read/write between the scene-lifecycle path (main) and the KVO path (arbitrary queue) becomes an unsynchronised race. Recommend isolating the flag (e.g. `@MainActor` on the view model or routing all reads/writes through a main-queue dispatch) before extending its use.
- `VODPlayerViewModel.swift:224` — `handleBackground()` reads `player.timeControlStatus` to decide whether the user was playing. `timeControlStatus` is mutated by AVFoundation off the main thread and is observed by the KVO callback at line 170. There is no synchronisation between this read and the KVO write path. In practice this read is usually consistent enough, but the captured value can lag a recent transition. Combined with the BLOCKER, this means `wasPlayingBeforeBackground` may be set to `false` for a player that was, in fact, mid-play — making the new guard occasionally fail open or fail closed depending on timing.
- `VODPlayerViewModel.swift:174-177` — The `timeControlStatus` observer clears `isPaused = false` on any non-`(.playing → .paused)` transition. When the OS auto-pauses during background and the status observer re-triggers play on foreground (per the BLOCKER), this resets `isPaused` to `false` and hides the pause overlay, so the user sees the video resuming with no visible pause state. Even after the BLOCKER is fixed, confirm `isPaused` reflects user intent rather than just transport state, otherwise the pause UI will lie during the bug window.
- `VODPlayerViewModel.swift:228-236` — `handleForeground()` does not clear `wasPlayingBeforeBackground` after acting on it. If `handleForeground()` is called twice without an intervening `handleBackground()` (e.g. duplicate scenePhase deliveries, sheet dismissal flows that re-fire foreground), the second call will re-issue `player.play()` even if the user paused between the two foreground events. Reset the flag to `false` after consuming it.

### 🟢 NICE-TO-CHECK
- `VODPlayerViewModel.swift:108` — The resolution-fetch failure branch calls `self?.player?.play()`. If this completes after the user has paused (including after a background round-trip), it starts playback regardless of `isPaused` / `wasPlayingBeforeBackground`. Adjacent to the BLOCKER class of bug — worth confirming this path is gated correctly.
- `VODPlayerViewModel.swift:114-117` — `stopPlayer()` nils the player but does not call `removeObservers()`. Observers in `observations` capture the old `player` directly (`player.observe` and the `player` closure capture in the `.readyToPlay` arm). After `stopPlayer()` plus a subsequent `loadPlaybackUrl()`, stale observers may still be alive against the previous AVPlayer instance until `deinit`. Not part of the current change, but the auto-play observer is the same observer chain implicated above.
- `VODPlayerViewModel.swift:155` — The status observer closure captures `player` strongly (the function parameter), independent of `self`. This keeps the AVPlayer alive for the lifetime of the observation; combined with the `stopPlayer()` issue above, it can keep an "old" player playing audio in the background if `UIBackgroundModes` includes `audio`. Worth verifying background modes for this target.
- The file references `AVPlayerView.updateUIViewController` semantics (from the worked example in the skill), which are not visible in this fixture — if `AVPlayerViewController` retains the old AVPlayer across a `stopPlayer()` → `loadPlaybackUrl()` cycle, the fix still won't catch a "ghost" player that the controller is driving. Confirm in the SwiftUI wrapper.

### ✅ Cleared
- `handleBackground()` / `handleForeground()` themselves: the flag is set on the right transition (pre-pause) and the guard short-circuits when the player was paused before backgrounding. Logic is internally consistent for the single-lifecycle-event case.
- No new `Task`, `async`, `await`, actor, or `Sendable` surface introduced by the change itself — Step 4 concern is on *existing* KVO observers, not new async work.
- `pause()` (line 209-212) still calls `player?.pause()` directly and is unaffected by the new flag.
- `deinit` ordering (line 242-245): `removeObservers()` runs before `stopPlayer()`, so KVO observers can't fire against a half-torn-down player due to the change.
